# Spring Security와 JWT - Stateless 인증 방식 완벽 이해

이전 글에서는 Spring Security가 **Security Filter Chain**을 중심으로 인증(Authentication)과 인가(Authorization)를 처리하는 과정을 살펴보았습니다.

이번 글에서는 최근 REST API에서 가장 많이 사용되는 **JWT(Json Web Token)** 기반 인증 방식을 알아보고, Spring Security와 어떻게 연동되어 **Stateless 인증**을 구현하는지 살펴보겠습니다.

---

# 왜 JWT를 사용할까?

전통적인 웹 애플리케이션은 **세션(Session)** 기반 인증을 사용했습니다.

로그인에 성공하면 서버는 사용자의 세션을 생성하고, 브라우저는 세션 ID가 담긴 쿠키를 저장합니다.

이후 요청이 들어오면 서버는 세션 ID를 조회하여 로그인 여부를 확인합니다.

```text
로그인

↓

Session 생성

↓

Session ID 저장

↓

브라우저 Cookie 저장

↓

API 요청

↓

Session 조회

↓

사용자 확인
```

이 방식은 구현이 단순하지만 서버가 세션을 저장해야 한다는 단점이 있습니다.

여러 대의 서버를 운영하는 환경에서는 세션을 공유하거나 별도의 세션 저장소를 구성해야 합니다.

---

# Stateless 인증이란?

JWT 인증은 **Stateless(무상태)** 방식을 사용합니다.

Stateless란 **서버가 클라이언트의 로그인 상태를 저장하지 않는 것**을 의미합니다.

사용자의 인증 정보는 서버가 아닌 **JWT 토큰**에 포함됩니다.

```text
Client

↓

로그인

↓

JWT 발급

↓

Client 저장

↓

API 요청

↓

JWT 검증

↓

응답
```

서버는 토큰만 검증하면 되므로 별도의 세션 저장소가 필요하지 않습니다.

---

# JWT란?

JWT(Json Web Token)는 사용자 정보를 안전하게 전달하기 위한 **토큰(Token)** 형식입니다.

JWT는 점(`.`)으로 구분된 세 부분으로 구성됩니다.

```text
Header.Payload.Signature
```

예를 들어

```text
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
eyJzdWIiOiJraW0iLCJyb2xlIjoiVVNFUiJ9
.
fS9mP...
```

각 부분의 역할은 다음과 같습니다.

| 구성 요소     | 설명             |
| --------- | -------------- |
| Header    | 토큰 타입과 서명 알고리즘 |
| Payload   | 사용자 정보(Claim)  |
| Signature | 위변조 검증을 위한 서명  |

---

# JWT의 구조

## Header

Header에는 토큰의 메타데이터가 저장됩니다.

```json
{
  "alg": "HS256",
  "typ": "JWT"
}
```

* `alg`: 서명 알고리즘
* `typ`: 토큰 타입

---

## Payload

Payload에는 사용자 정보가 저장됩니다.

```json
{
  "sub": "kim",
  "role": "USER",
  "iat": 1710000000,
  "exp": 1710003600
}
```

대표적인 Claim은 다음과 같습니다.

| Claim | 설명     |
| ----- | ------ |
| sub   | 사용자 ID |
| role  | 권한     |
| iat   | 발급 시간  |
| exp   | 만료 시간  |

> **주의:** Payload는 Base64로 인코딩될 뿐 암호화되지 않습니다. 따라서 비밀번호, 주민등록번호, 카드 정보 등 민감한 데이터는 절대로 JWT에 저장해서는 안 됩니다.

---

## Signature

Signature는 토큰이 위변조되지 않았는지 검증합니다.

```
HMACSHA256(
    Base64(Header) + "." + Base64(Payload),
    SecretKey
)
```

Payload가 변경되면 Signature도 달라지므로 서버는 토큰의 무결성을 확인할 수 있습니다.

---

# 로그인 과정

JWT 인증 과정은 다음과 같습니다.

```text
Client

↓

로그인 요청

↓

Spring Security

↓

UserDetailsService

↓

Database

↓

사용자 인증

↓

JWT 생성

↓

Client 반환
```

로그인에 성공하면 서버는 JWT를 생성하여 클라이언트에게 전달합니다.

---

# 이후 API 요청

로그인 이후에는 아이디와 비밀번호를 다시 보내지 않습니다.

대신 JWT를 요청 헤더에 포함합니다.

```http
GET /users

Authorization: Bearer eyJhbGciOi...
```

요청 흐름은 다음과 같습니다.

```text
Request

↓

Authorization Header

↓

JWT Filter

↓

토큰 검증

↓

Controller
```

---

# Spring Security와 JWT

JWT 인증은 대부분 **Filter**에서 처리됩니다.

```text
HTTP Request

↓

JWT Authentication Filter

↓

토큰 검증

↓

SecurityContext 저장

↓

DispatcherServlet

↓

Controller
```

JWT Filter는 `Authorization` 헤더를 확인하고, 유효한 토큰이라면 사용자 정보를 `SecurityContext`에 저장합니다.

---

# JWT Filter 예시

JWT Filter는 보통 `OncePerRequestFilter`를 상속하여 구현합니다.

```java
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain)
            throws ServletException, IOException {

        String token = resolveToken(request);

        if (token != null && jwtProvider.validate(token)) {

            Authentication authentication =
                    jwtProvider.getAuthentication(token);

            SecurityContextHolder
                    .getContext()
                    .setAuthentication(authentication);
        }

        filterChain.doFilter(request, response);
    }

}
```

핵심 역할은 다음과 같습니다.

1. Authorization Header 확인
2. JWT 추출
3. 토큰 검증
4. Authentication 생성
5. SecurityContext 저장
6. 다음 Filter로 요청 전달

---

# SecurityContext에 저장되는 이유

JWT를 검증한 후에는 사용자 정보를 `SecurityContext`에 저장합니다.

```text
JWT 검증

↓

Authentication 생성

↓

SecurityContext 저장

↓

Controller
```

이후에는 Controller에서 현재 로그인한 사용자를 쉽게 조회할 수 있습니다.

```java
Authentication authentication =
        SecurityContextHolder
                .getContext()
                .getAuthentication();
```

또는

```java
@GetMapping("/me")
public UserDto me(Authentication authentication) {

    return ...

}
```

처럼 `Authentication`을 파라미터로 직접 받을 수도 있습니다.

---

# SecurityFilterChain 설정

JWT 인증을 사용하는 경우 세션을 사용하지 않도록 설정하는 것이 일반적입니다.

```java
@Bean
SecurityFilterChain filterChain(HttpSecurity http) throws Exception {

    http
        .csrf(csrf -> csrf.disable())
        .sessionManagement(session ->
            session.sessionCreationPolicy(
                SessionCreationPolicy.STATELESS
            )
        )
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/login").permitAll()
            .anyRequest().authenticated()
        );

    return http.build();
}
```

`SessionCreationPolicy.STATELESS`를 설정하면 Spring Security는 세션을 생성하거나 사용하지 않습니다.

---

# JWT의 장점

JWT 기반 인증은 다음과 같은 장점을 제공합니다.

* 서버가 세션을 저장하지 않아도 된다.
* 서버를 여러 대로 확장하기 쉽다.
* 모바일 앱과 SPA(React, Vue 등)에 적합하다.
* 인증 정보를 표준화된 형식으로 전달할 수 있다.
* 마이크로서비스 환경에서 활용하기 쉽다.

---

# JWT의 단점

반면 JWT에는 고려해야 할 점도 있습니다.

* 토큰이 탈취되면 만료 전까지 사용할 수 있다.
* 발급된 토큰을 즉시 무효화하기 어렵다.
* Payload에 민감한 정보를 저장할 수 없다.
* 토큰 크기가 세션 ID보다 크다.

이러한 문제를 해결하기 위해 실제 서비스에서는 다음과 같은 전략을 함께 사용합니다.

* Access Token + Refresh Token
* 짧은 Access Token 만료 시간
* Refresh Token 저장소 관리
* HTTPS 사용
* 토큰 재발급 정책

---

# 전체 요청 흐름

Spring Security와 JWT를 함께 사용한 요청 흐름은 다음과 같습니다.

```text
Client
   │
로그인
   │
   ▼
Spring Security
   │
사용자 인증
   │
JWT 발급
   │
Client 저장
   │
────────────────────────────
API 요청
Authorization: Bearer Token
   │
   ▼
JWT Filter
   │
토큰 검증
   │
Authentication 생성
   │
SecurityContext 저장
   │
DispatcherServlet
   │
Controller
   │
Service
   │
Repository
   │
Database
   │
JSON Response
```

# 마무리

JWT는 서버가 로그인 상태를 저장하지 않는 **Stateless 인증 방식**을 구현하기 위한 대표적인 기술입니다. 사용자는 로그인 후 발급받은 토큰을 요청마다 함께 전송하고, 서버는 해당 토큰의 유효성을 검증한 뒤 인증 정보를 `SecurityContext`에 저장하여 요청을 처리합니다.

Spring Security와 JWT를 함께 사용하면 확장성이 뛰어난 REST API를 구축할 수 있으며, 특히 모바일 애플리케이션, SPA(React, Vue), 마이크로서비스 아키텍처에서 널리 활용됩니다.

다만 JWT는 만능 해결책이 아닙니다. 토큰 탈취, 만료 정책, 재발급 전략, 로그아웃 처리 등을 함께 설계해야 안전한 인증 시스템을 구축할 수 있습니다.

다음 글에서는 **Access Token과 Refresh Token의 차이**, **토큰 재발급 전략**, 그리고 **실무에서 JWT를 안전하게 운영하는 방법**을 중심으로 살펴보겠습니다.
