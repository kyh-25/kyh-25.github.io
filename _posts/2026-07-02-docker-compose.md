## docker run만으로 충분하지 않은 순간

지금까지는 컨테이너 하나를 `docker run`으로 띄우는 상황만 다뤘습니다. 하지만 실제 애플리케이션은 대부분 컨테이너 여러 개가 함께 동작합니다.

예를 들어 Spring Boot 애플리케이션 하나를 제대로 띄우려면 최소한 다음이 필요합니다.

- 애플리케이션 컨테이너
- 데이터베이스 컨테이너 (예: PostgreSQL)
- (경우에 따라) 캐시 컨테이너 (예: Redis)

이걸 전부 `docker run`으로 직접 실행한다고 생각해보겠습니다.

```bash
docker network create my-net

docker run -d --name my-db --network my-net \
  -e POSTGRES_PASSWORD=1234 \
  postgres:16

docker run -d --name my-app --network my-net \
  -p 8080:8080 \
  -e SPRING_DATASOURCE_URL=jdbc:postgresql://my-db:5432/postgres \
  my-app:1.0
```

당장은 두 컨테이너뿐이라 버틸 만해 보이지만, 여기엔 이미 여러 문제가 숨어 있습니다.

- 네트워크를 먼저 수동으로 만들어야 하고, 두 컨테이너 모두 같은 네트워크에 연결해야 합니다.
- 옵션(`-e`, `-p`, `--name` 등)이 컨테이너마다 계속 늘어나고, 매번 같은 명령어를 순서대로 정확히 입력해야 합니다.
- 컨테이너를 껐다 켤 때도, 지울 때도 이 긴 명령어들을 하나씩 반복해야 합니다.
- 팀원에게 "이렇게 실행하세요"라고 설명하려면 명령어 목록을 따로 문서로 정리해야 합니다.

컨테이너가 3개, 5개로 늘어나면 이 방식은 곧 감당하기 어려워집니다.

## 그냥 앱과 DB를 이미지 하나에 합치면 안 될까?

여기서 자연스럽게 드는 생각이 있습니다. "컨테이너를 여러 개 관리하기 귀찮으면, 애초에 앱과 DB를 한 이미지 안에 다 넣어버리면 되지 않을까?"

기술적으로 불가능하지는 않지만, 다음과 같은 이유로 거의 쓰이지 않는 방식입니다.

- **생명주기가 다릅니다.** 애플리케이션 코드는 하루에도 여러 번 재배포되지만, 데이터베이스는 계속 떠 있어야 합니다. 하나로 합치면 앱을 새 버전으로 배포할 때마다 DB 프로세스까지 같이 재시작/재생성되어 데이터가 유실될 위험이 생깁니다.
- **독립적으로 확장할 수 없습니다.** 트래픽이 늘어 앱 컨테이너를 여러 개로 늘리고 싶어도, DB가 앱 안에 박혀 있으면 DB도 같이 여러 개가 되어 데이터가 흩어져버립니다. 앱은 여러 개, DB는 하나만 공유하는 흔한 구조를 만들 수 없습니다.
- **Docker의 도구들이 "컨테이너 하나 = 책임 하나"를 전제로 설계돼 있습니다.** `docker logs`, 헬스체크, 재시작 정책이 모두 컨테이너 단위로 동작하기 때문에, 프로세스를 억지로 합치면 이 기능들을 제대로 활용할 수 없습니다.

그래서 "컨테이너 하나에는 프로세스(책임) 하나"가 관례이고, 여러 개의 단일 책임 컨테이너를 네트워크로 엮어 하나의 애플리케이션처럼 동작하게 만드는 역할을 Compose가 담당합니다.

## Docker Compose란?

**Docker Compose**는 여러 컨테이너로 이루어진 애플리케이션을 **하나의 YAML 파일**로 정의하고, 명령어 한 줄로 전체를 실행·종료할 수 있게 해주는 도구입니다.

방금 본 `docker run` 두 번을 Compose로 옮기면 이렇게 됩니다.

```yaml
# docker-compose.yml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: 1234

  app:
    image: my-app:1.0
    ports:
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://db:5432/postgres
    depends_on:
      - db
```

이제 아래 명령어 하나로 두 컨테이너가 동시에 뜹니다.

```bash
docker compose up -d
```

네트워크를 따로 만들지 않았다는 점에 주목해야 합니다. Compose가 이 파일을 실행하는 순간, 프로젝트 전용 네트워크를 **자동으로** 만들어서 `db`, `app` 두 서비스를 그 안에 연결해줍니다.

## docker-compose.yml의 기본 구조

방금 본 파일을 한 줄씩 뜯어보겠습니다.

```yaml
services:        # 이 애플리케이션을 구성하는 컨테이너들
  db:            # 서비스 이름 (컨테이너 이름이자, 네트워크 안에서의 호스트명이 됨)
    image: postgres:16       # 사용할 이미지
    environment:             # 컨테이너에 주입할 환경변수
      POSTGRES_PASSWORD: 1234

  app:
    image: my-app:1.0
    ports:                   # 호스트 포트:컨테이너 포트
      - "8080:8080"
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://db:5432/postgres
    depends_on:               # app보다 db를 먼저 시작
      - db
```

| 항목 | 역할 |
| --- | --- |
| `services` | 실행할 컨테이너들을 정의하는 최상위 항목 |
| 서비스 이름(`db`, `app`) | 컨테이너 이름이자, 다른 서비스에서 이 컨테이너를 찾을 때 쓰는 호스트명 |
| `image` | 사용할 이미지 (직접 만든 Dockerfile이 있다면 `build: .`로 대체 가능) |
| `ports` | 호스트에 노출할 포트 |
| `environment` | 컨테이너 내부에 주입되는 환경변수 |
| `depends_on` | 시작 순서 지정 (다른 서비스가 먼저 뜬 *뒤에* 시작) |

> ❓ **생각해보기**
> `SPRING_DATASOURCE_URL`에 `localhost`가 아니라 `db`라는 이름이 들어가 있습니다. `app` 컨테이너 입장에서 `db`는 자기 자신이 아닌 완전히 다른 컨테이너인데, 어떻게 `db`라는 이름만으로 접속할 수 있는 걸까요? 이전 글에서 다룬 "컨테이너는 격리된 프로세스"라는 개념과 연결해서 먼저 생각해보세요.
{: .prompt-warning }

## 서비스 이름으로 통신이 되는 이유

이전 글에서 컨테이너는 Namespace로 격리된 프로세스라고 했는데, 여기엔 **네트워크 네임스페이스**도 포함됩니다. 즉 컨테이너마다 자기만의 독립된 네트워크 스택(자기만의 `localhost`, 자기만의 포트 공간)을 가지고 있어서, `app` 컨테이너의 `localhost`와 `db` 컨테이너의 `localhost`는 완전히 다른 공간입니다. 격리되어 있으니 기본적으로는 서로의 존재조차 알 수 없습니다.

```text
app 컨테이너의 네트워크 네임스페이스        db 컨테이너의 네트워크 네임스페이스
   (자기만의 localhost)                      (자기만의 localhost)
        │                                          │
        └──────────── 기본적으로 서로 분리됨 ────────┘
```

네트워크가 갑자기 등장하는 이유가 여기 있습니다. 네트워크도 격리 대상 중 하나이기 때문에, 이 격리를 넘어 컨테이너끼리 통신하려면 **둘을 이어주는 별도의 네트워크**가 필요합니다. `docker compose up`을 실행하면 Compose는 이 프로젝트만을 위한 **전용 네트워크**를 하나 만들고, 정의된 모든 서비스를 그 네트워크에 자동으로 연결합니다. 이 네트워크는 각 컨테이너의 네트워크 네임스페이스를 가상 이더넷으로 가상 스위치에 연결해, 서로 다른 네임스페이스에 있는 두 프로세스가 IP로 통신할 수 있도록 다리를 놓아주는 역할을 합니다.

```text
docker compose up
      │
      ▼
프로젝트 전용 네트워크 생성
      │
      ▼
services에 정의된 컨테이너들을 모두 이 네트워크에 연결
      │
      ▼
같은 네트워크 안에서는 서비스 이름 = 호스트명(DNS)
```

이 네트워크 안에서는 Docker가 내장 DNS 역할을 해서, 서비스 이름(`db`)을 요청하면 실제 `db` 컨테이너의 IP로 자동 변환해줍니다. 그래서 `app` 컨테이너 입장에서는 `db`라는 이름이 곧 데이터베이스 컨테이너의 주소가 됩니다.

반대로 이 네트워크에 속하지 않은 컨테이너, 즉 `docker run`으로 따로 띄운 컨테이너는 `db`라는 이름을 몰라 접속할 수 없습니다. `docker run`으로 직접 네트워크를 만들어 연결해야 했던 이유가 바로 이것이었고, Compose는 이 과정을 파일 하나로 자동화해준 것입니다.

여기서 `ports`와 이 네트워크는 서로 다른 층위라는 점을 구분해야 합니다.

- `ports: - "8080:8080"`(포트 발행)은 **Docker 네트워크 바깥**, 즉 호스트 머신이나 브라우저처럼 Compose 네트워크에 속하지 않은 존재가 컨테이너에 접근할 때 필요한 통로입니다.
- 반면 `app`이 `db:5432`로 접속하는 것은 **같은 Compose 네트워크 안에서 컨테이너끼리 직접 통신**하는 것이라 포트 발행이 필요 없습니다. 실제로 위 예시에서 `db` 서비스에는 `ports`가 전혀 없는데도 `app`은 `db:5432`로 정상 접속합니다. 컨테이너 내부 포트(5432)로 네트워크 안에서 바로 통신하기 때문입니다.

즉 `ports`는 "바깥에서 안으로 들어오는 문"이고, 서비스 이름 기반 통신은 "안에서 안끼리 연결하는 통로"로, 서로 목적이 다릅니다.

## 자주 쓰는 Compose 명령어

```bash
docker compose up -d       # 모든 서비스를 백그라운드로 시작
docker compose ps           # 이 프로젝트의 컨테이너 상태 확인
docker compose logs -f app  # 특정 서비스 로그를 실시간으로 확인
docker compose stop         # 컨테이너는 남기고 정지만
docker compose down         # 컨테이너와 네트워크까지 정리
```

> 🛠 **직접 해보기**
> 위 `docker-compose.yml` 예시(이미지 이름은 원하는 걸로 바꿔도 됩니다)를 파일로 저장하고 `docker compose up -d`를 실행해보세요. 이어서 `docker compose ps`로 두 컨테이너가 모두 `Up` 상태인지 확인하고, `docker compose logs -f app`으로 로그를 살펴보세요.

## 마무리

Docker Compose는 새로운 격리 기술이 아니라, 여러 컨테이너를 하나의 YAML 파일로 정의하고 네트워크 연결까지 자동으로 처리해주는 **오케스트레이션 도구**입니다. `docker run`을 여러 번 반복하며 네트워크를 직접 관리하던 문제를, `services` 아래 각 컨테이너를 선언하는 방식으로 해결합니다.

같은 Compose 네트워크 안에서는 서비스 이름이 곧 호스트명이 되어 컨테이너 간 통신이 가능해집니다.

로컬 개발 환경에서 여러 컨테이너를 함께 다뤄야 한다면, Compose는 사실상 기본 도구입니다. 다만 여러 대의 서버에 걸쳐 컨테이너를 운영해야 하는 프로덕션 규모에서는 Compose만으로는 한계가 있어 Kubernetes 같은 오케스트레이션 도구를 사용하게 됩니다.
